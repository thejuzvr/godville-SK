# План улучшения проекта Godville-SK

## Проблема 1: Мысли и поток отчётов в журнал приключений

### 1.1 Улучшение содержания мыслей
**Файлы:** `lib/godville_sk/game_data.ex:599-648`, `lib/godville_sk/hero/state_machine.ex:834-837`

**Текущее поведение:** Мысли - короткие односложные фразы без повествовательного контекста.

**План:**
- Расширить мысли в `GameData.thoughts()` - добавить более длинные, повествовательные варианты
- Добавить контекстуальные мысли (мысль зависит от текущей локации, уровня, состояния)
- В `state_machine.ex:think()` добавить логику выбора мысли в зависимости от контекста:
  - После боя - размышления о победе/поражении
  - В подземелье - атмосферные описания
  - В городе - мысли о торговле/отдыхе

### 1.2 Замедление потока тиков
**Файлы:** `lib/godville_sk/hero.ex:11`, `lib/godville_sk/hero/state_machine.ex:19-26`

**Текущее поведение:** Tick каждые 2 секунды, все действия выполняются с этой скоростью.

**План:**
- Изменить интервал тика в `hero.ex:11`: `@tick_interval` сделать динамическим вместо фиксированных 2 сек
- Добавить механизм "задержки действий" в `StateMachine.tick()`:
  - Для мыслей: random 5-120 сек (добавить счётчик тиков до следующей мысли)
  - Для боёв: увеличить паузу между раундами на 15%
  - Для заданий: добавить задержку 60-300 сек между этапами квеста
- Реализовать через поле `next_action_at` в state и проверку времени в tick

### 1.3 Пустые/неиспользуемые предметы
**Файлы:** `lib/godville_sk/hero/items.ex:234-236`, `lib/godville_sk/game/items.ex:99-118`

**Текущее поведение:** Функция `find_item` ищет по всем определённым предметам, а не по конкретным в инвентаре героя. Предметы в инвентаре хранятся как строки (имена), а не структуры.

**План:**
- Инвентарь хранит имена предметов (строки), нужно правильно сопоставлять с определениями
- Исправить `find_item` - искать среди всех определений предметов по имени
- Проверить `GameContent.get_random_loot()` - возвращает ли структуру предмета
- Добавить отладку для понимания какие именно предметы "пустые"

## Проблема 2: Исправление мелких визуальных проблем

### 2.1 Перекрытие панелей (задание боем)
**Файлы:** `lib/godville_sk_web/live/dashboard_live.ex:219`, `dashboard_live.ex:280`

**Текущее поведение:** 
- Панель боя показывается при `status in [:combat, :combat_initiative, :combat_round]`
- Панель заданий показывается при `status == :questing`
- Но во время боя target может быть и монстр и квест одновременно

**План:**
- Изменить условие отображения панели заданий: показывать только когда статус именно `:questing`, а не во время боя
- В `dashboard_live.ex:59` изменить условие:
  ```
  <%= if @hero_state.status == :questing && @hero_state.target && !is_combat_status(@hero_state.status) do %>
  ```
- Добавить функцию `is_combat_status(status)` для проверки боевых статусов

### 2.2 Статус "Бездействует" во время боя
**Файлы:** `lib/godville_sk_web/live/dashboard_live.ex:778-798`

**Текущее поведение:** Функция `status_text/1` не обрабатывает `:combat_initiative` и `:combat_round`, они попадают в fallback case и показывают "Бездействует".

**План:**
- Добавить обработку статусов боя в `status_text`:
  ```elixir
  defp status_text(:combat_initiative), do: "Вступает в бой"
  defp status_text(:combat_round), do: "В бою"
  ```
- Аналогично обновить `status_class` для боевых статусов

### 2.3 Нарушение логики заданий (этапов больше положенных)
**Файлы:** `lib/godville_sk/hero/state_machine.ex:311-331`, `hero.ex:39`

**Текущее поведение:** При начале квеста `quest_progress = 0`, но в каждом тике он инкрементируется без проверки, что квест ещё не завершён. Если во время выполнения квеста герой вступает в бой (смена статуса), а потом возвращается к квесту - прогресс продолжает расти.

**План:**
- В `handle_status(:questing, state)` добавить проверку: если квест уже завершён (progress >= steps), сбрасывать прогресс или завершать квест
- Добавить guard: не инкрементировать progress если уже достигнут максимум steps
- Возможно добавить защиту от "двойного тика" - добавить флаг `quest_tick_processed` в текущем тике
